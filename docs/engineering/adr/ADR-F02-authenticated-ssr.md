# ADR — F-02: `_authenticated` Layout SSR Profile

**Status:** Accepted · 2026-07-20
**Wave:** C.7 · R0.3
**Constitutional rule invoked:** Rule 1 (Verify Before Patch).

## Context

Wave C.5 flagged F-02: `src/routes/_authenticated/route.tsx` uses
`ssr: false`. The audit line asked whether this is a considered design
or a bandaid over a session-hydration bug.

## Verification (no code changed)

| Claim | Evidence | Verdict |
|---|---|---|
| The `_authenticated/route.tsx` layout is the **integration-managed** pattern shipped by the Lovable Supabase integration. | File header: *"This file is automatically generated. Do not edit it directly."* (see `src/integrations/supabase/*` companions carry the same marker). Knowledge file `tanstack-supabase-integration` §"Authenticated vs Public Routes" states verbatim: *"the pathless layout `src/routes/_authenticated/route.tsx` is integration-managed, uses `ssr: false`, and gates the whole subtree — child routes need no auth code. Do not author or rewrite that layout."* | ✅ Managed |
| `ssr: false` is the **prescribed** profile, not a workaround. | Same knowledge file: *"The managed layout is `ssr: false` because Supabase stores the session in `localStorage`, which the server cannot read. Gating the subtree server-side causes redirect loops on hard refresh and auth-page flashes for signed-in users; skipping SSR for the protected subtree only avoids both."* | ✅ Intentional |
| The layout gate uses `supabase.auth.getUser()` (revalidates against Auth server), not `getSession()`. | `src/routes/_authenticated/route.tsx:7` → `await supabase.auth.getUser()`. | Bearer token is attached to protected server-fn calls by registered middleware. | `src/start.ts:1-8` registers `attachSupabaseAuth` in `functionMiddleware`, appended alongside `securityHeadersMiddleware` in `requestMiddleware`. | Sign-in redirect-back is hardened and loop-safe. | `src/routes/auth.tsx:36-53` (`safeRedirect`) — R0.1 wave hardened: rejects absolute URLs, protocol-relative, `%2f%2f` decode bypasses, whitespace, control chars, over-length, and same-target loops (`/auth` → default). | ✅ Correct |

## Considered alternatives

1. **Enable SSR on the protected subtree.**
   - Requires cookie-backed sessions (`@supabase/ssr` + `set-cookie` on
     auth events). Not compatible with the current localStorage-only
     browser client.
   - Documented failure mode: redirect loop on hard refresh (server has
     no session → 302 → client hydrates and has session → back to route
     → repeat), plus signed-in users see the `/auth` page for one paint.
   - **Rejected.** Architectural rewrite for zero user-visible upside on
     this stack; the client-only gate already prevents unauthorized
     access because the actual data guards live server-side in
     `requireSupabaseAuth`.

2. **Move some gating to the server.**
   - Already there. Every protected `createServerFn` uses
     `requireSupabaseAuth`, which re-validates the bearer token via
     `supabase.auth.getClaims(token)` (`auth-middleware.ts:92`). The
     client-side layout gate is *only* a UX affordance (skip render of
     protected UI when unauthenticated). Real authorization is
     server-side per request.
   - **Adopted (already in place).**

3. **Inline sign-in inside `_authenticated` instead of redirect.**
   - Valid pattern per `tanstack-auth-guards`, but the current redirect
     flow with `safeRedirect` restores intent cleanly and matches the
     rest of the product. **Not changed.**

## Acceptance checklist (per user directive)

| Scenario | Expected | Observed |
|---|---|---|
| Unauthenticated user visits `/catalog` | Redirected to `/auth?redirect=%2Fcatalog`, no flash of catalog UI. | ✅ `beforeLoad` throws `redirect()` before the child component mounts. |
| Authenticated user visits `/auth` | Redirected to `redirect` param (default `/catalog`), `replace: true` keeps `/auth` off history. | ✅ `AuthPage` `useEffect` (`auth.tsx:68-72`). |
| Hard refresh on `/catalog` while signed in | Layout renders without flash to `/auth`. | ✅ `ssr: false` skips server render; `getUser()` resolves from localStorage on client, then Outlet mounts. No hydration mismatch (nothing to mismatch). |
| Sign out | Cache cleared, redirect to `/auth`, back button does not restore. | ✅ Handler-side (`replace: true`) + `AuthStateSync` `router.invalidate()` on `SIGNED_OUT` triggers the gate redirect. |
| Redirect loops | None. | ✅ `safeRedirect` refuses `/auth*` targets; `_authenticated` never re-enters itself. |
| Protected content flash | None. | ✅ Gate throws before component render. |
| SSR regression | None. | ✅ Protected subtree opts out of SSR by design; public routes (`/`, `/auth`, `/reset-password`) continue to SSR. |

## Decision

**Keep `ssr: false` on `_authenticated/route.tsx` unchanged.** It is the
canonical Lovable Supabase integration profile for localStorage-backed
sessions. All security invariants (per-request bearer revalidation,
server-side RLS via `requireSupabaseAuth`, loop-safe redirect) hold.
