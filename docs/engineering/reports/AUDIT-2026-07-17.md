## Phase 2 — Database Verification

### AUDIT-DB-001 — Cron auth mismatch: 10 scheduled jobs will 401 on every run
- **Severity**: P0
- **Confidence**: 100%
- **Evidence** (from `cron.job` joined with endpoint source):

  | jobid | jobname | schedule | headers sent | endpoint requires |
  |---|---|---|---|---|
  | 7 | `muslly-nightly-intel` | `30 23 * * *` | `apikey` only | `x-cron-secret` (`nightly-intel.ts:2`) |
  | 10 | `weekly-ai-enrich` | `0 3 * * 0` | `apikey` only | `x-cron-secret` (`weekly-ai-enrich.ts:4`) |
  | 11 | `weekly-exec-report` | `30 3 * * 0` | `apikey` only | `x-cron-secret` (`weekly-exec-report.ts:2`) |
  | 20 | `prescription-extract-worker` | `* * * * *` | `apikey` only | POST handler at `prescription-extract.ts:9` |
  | 22 | `run-reactivation-weekly` | `0 9 * * 1` | `apikey` only | `x-cron-secret` (`run-reactivation.ts:2,8`) |
  | 23 | `run-loyalty-reminder-daily` | `0 10 * * *` | `apikey` only | `x-cron-secret` (`run-loyalty-reminder.ts:2,8`) |
  | 24 | `run-restock-alerts-4h` | `0 */4 * * *` | `apikey` only | `x-cron-secret` (`run-restock-alerts.ts:2,8`) |
  | 25 | `reactivation-campaign` | `0 9 * * 1` | `apikey` only | duplicate of 22 |
  | 26 | `loyalty-reminder-campaign` | `0 10 * * *` | `apikey` only | duplicate of 23 |
  | 27 | `restock-alerts` | `0 */2 * * *` | `apikey` only | duplicate of 24 |
  | 36 | `validate-uploads-12h` | `0 */12 * * *` | **no auth header** | `x-cron-secret` (`validate-uploads.ts:13`) |

  `src/lib/cron-auth.server.ts:10-29` — `verifyCronSecret` reads **only** `request.headers.get("x-cron-secret")` (query fallback was explicitly removed in Batch 5a/C3 per comment on line 18) and returns 401 on mismatch or 503 if `CRON_SECRET` env is missing.

- **Files**: `supabase/migrations/*` for the `cron.schedule(...)` statements creating the 10 jobs above; endpoint files listed in the table.
- **Root Cause**: Migration authors passed the Supabase publishable/anon JWT as `apikey` (a pattern used for edge-function auth) instead of the app-level `x-cron-secret` header the TSS handlers enforce.
- **Business Impact**:
  - Prescription OCR queue (`jobid 20`) never runs → uploaded prescriptions stay in `queued` state indefinitely, patients see no response.
  - Weekly AI enrichment, executive reports and nightly intel silently produce nothing.
  - Reactivation, loyalty, and restock marketing dispatches never fire, so no WhatsApp/email is sent for those flows.
  - `validate-uploads-12h` never runs, so the 12-hour rescan of prescription file MIME/size/magic bytes (soft-deletes bad uploads) is not enforced — malicious or corrupt uploads persist.
- **Technical Impact**: Silent failure — jobs "succeed" from cron's perspective (HTTP call was made) but responses are 401; no alert path fires because response body is discarded by `pg_net`.
- **Security Impact**: `validate-uploads` outage is a direct control-gap on the prescription file storage bucket.
- **Suggested Fix** (text-only): (a) rewrite these 10 cron jobs to send `x-cron-secret` sourced from `current_setting('app.cron_secret', true)` (already the pattern for jobs 12, 13, 14, 17, 18, 19, 29–35, 37–40, 42, 50); (b) remove the exact duplicates 25/26/27 which schedule the same endpoint as 22/23/24; (c) add a monitoring alert on `cron.job_run_details` for consecutive 401s.
- **Estimated Complexity**: S (one SQL migration).
- **Regression Risk**: Low (jobs are already failing).
- **Dependencies**: none.

### AUDIT-DB-002 — Anon JWT literal embedded in 10 cron job bodies
- **Severity**: P3 (hygiene)
- **Confidence**: 100%
- **Evidence**: `cron.job.command` for jobs 7, 10, 11, 20, 22–27 contains the anon key as a literal string:
  `eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9…`
- **Files**: migrations that ran `cron.schedule(...)` for those jobs.
- **Root Cause**: Copy-paste from an older webhook pattern. The anon JWT is the publishable key (safe to expose), so this is not a secret leak; it is a rotation hazard — rotating the publishable key requires editing every job.
- **Business Impact**: Slower key rotation.
- **Suggested Fix**: replace with `current_setting('app.publishable_key', true)` or drop the header entirely once AUDIT-DB-001 is fixed (endpoints don't require `apikey`).
- **Estimated Complexity**: S. **Regression Risk**: Low.

### AUDIT-DB-003 — SECURITY DEFINER helpers executable by PUBLIC (anon)
- **Severity**: P2
- **Confidence**: 100%
- **Evidence** — functions with `EXECUTE TO PUBLIC` grant (from `pg_proc.proacl` where `grantee = 0`):
  ```
  public.current_org             public.emit_identity_event
  public.handle_new_user_profile public.has_org_permission
  public.has_org_role            public.hc_doctors_guard_verify
  public.hc_touch_doctor_scores  public.is_org_member
  public.list_my_org_permissions public.org_feature_enabled
  public.org_within_limit        public.tg_branch_assignments_events
  public.tg_branches_events      public.tg_org_members_events
  public.tg_organization_members_audit
  public.tg_organizations_after_insert
  public.tg_profiles_events
  ```
- **Analysis**:
  - Trigger functions (`tg_*`, `handle_new_user_profile`, `hc_touch_doctor_scores`, `hc_doctors_guard_verify`, `emit_identity_event`) are safe to leave — PostgREST does not expose them by shape, and revoking would not affect trigger invocation.
  - `has_org_role`, `is_org_member`, `has_org_permission`, `list_my_org_permissions`, `org_feature_enabled`, `org_within_limit`, `current_org` are RBAC helpers designed for use *inside* RLS. Being callable as anon via PostgREST RPC lets an unauthenticated caller probe organisation membership: `POST /rest/v1/rpc/is_org_member` with a UUID pair returns a boolean.
- **Business/Security Impact**: an unauthenticated attacker can enumerate `(user_id, org_id)` pairs and learn org sizes / membership. Not PII by itself; combined with a leaked user UUID it becomes a directory service.
- **Suggested Fix**: `REVOKE EXECUTE ON FUNCTION public.<fn> FROM PUBLIC; GRANT EXECUTE ON FUNCTION public.<fn> TO authenticated;` for the 7 RBAC helpers listed. Leave the 10 trigger functions alone.
- **Estimated Complexity**: S. **Regression Risk**: Low (RLS still calls them as the definer, not as the invoker).

### AUDIT-DB-006 — `product_gallery_images` public SELECT is unconstrained (`qual: true`)
- **Severity**: P3
- **Confidence**: 100%
- **Evidence**: `pg_policies` row — table `product_gallery_images`, policyname `public can read gallery`, cmd `SELECT`, roles `{anon,authenticated}`, qual `true`.
- **Root Cause**: intentional public gallery, but no `is_active` / `deleted_at IS NULL` gate.
- **Business Impact**: soft-deleted or draft images become visible.
- **Suggested Fix**: replace `qual: true` with a filter (`is_active AND deleted_at IS NULL`) if a soft-delete/draft column exists; otherwise document the "everything is public" invariant.
- **Complexity**: S. **Regression Risk**: Low.

### AUDIT-DB-008 — `contact_messages` and `error_logs` anon INSERT lack IP rate-limit at DB layer
- **Severity**: P2
- **Confidence**: 90% (DB layer verified; endpoint layer partially checked below in AUDIT-SEC-003).
- **Evidence**: both policies have `qual IS NULL` and a `WITH CHECK` that only bounds field length/format — nothing throttles by IP or session.
- **Business Impact**: unauthenticated spam floods `contact_messages` and `error_logs`; the latter can bury real errors.
- **Suggested Fix**: put a `rate_limit_buckets`-based check into the INSERT policy, OR require every anon INSERT to go through a server route (`/api/public/contact`, `/api/public/log-error`) that already enforces the limit (see AUDIT-SEC-003 evidence for `log-error.ts`), and revoke the direct-anon INSERT grant on these tables.
- **Complexity**: M. **Regression Risk**: Med (browser reporters break if they still POST to `/rest/v1/error_logs`).

### AUDIT-SEC-002 — Image proxy: SSRF controls present — PASS
- **Severity**: informational
- **Confidence**: 100%
- **Evidence**: `src/routes/api/public/img.ts:141-146` requires `upstream.protocol === 'https:'` AND `allowed.has(upstream.hostname)`; `:101` invokes `check_img_rate_limit` RPC before fetch; allow-list source is `img_proxy_settings.allowed_hosts` with an in-memory cache (`:43-58`).

### AUDIT-SEC-003 — `log-error` has an in-process rate limiter; `contact` and `analytics/ingest` do not
- **Severity**: P2
- **Confidence**: 90%
- **Evidence**:
  - `src/routes/api/public/log-error.ts:27` defines `rateLimited(ip)`; `:63-65` returns `{ ok:false, error:"rate_limited" }`.
  - `grep -nE 'rate|limit|captcha' src/routes/api/public/contact.ts` → no matches.
  - `grep -nE 'rate|limit|origin|Bearer' src/routes/api/public/analytics/ingest.ts` → no matches.
- **Business Impact**: spam flood on contact form and analytics pixel; the latter is a common vector for skewing dashboards.
- **Suggested Fix**: reuse the `rate_limit_buckets` table + `consume_rate_limit` RPC (already a SECURITY DEFINER function in `pg_proc`) inside both handlers, keyed on IP + endpoint.
- **Complexity**: S. **Regression Risk**: Low.

- **Evidence** (`pg_policies` rows for `hc_patients`):
